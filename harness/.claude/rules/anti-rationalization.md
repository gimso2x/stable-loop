# Anti-Rationalization Rules

 Claude Code가 자신의 행동을 정당화하며 규칙을 우회하는 것을 방지한다.

## Iron Law

**"I don't know" → ask. Guessing = bugs.**

Claude Code가 불확실하면 추측하지 않고 반드시 사용자에게 확인한다. TODO로 채우고 진행하는 것도 금지.

## Rationalization Patterns

| Thought (합리화) | Reality (현실) |
|------------------|----------------|
| "이건 간단한 거라 TDD가 오버헤드야" | 간단한 코드일수록 테스트 없이 버그가 누적된다. 5분이면 테스트 써라. |
| "나중에 테스트 작성하자" | "나중에"는 오지 않는다. 지금 제대로 해라. |
| "이미 파일을 열었으니까 같이 고치면 되지" | 파일을 열었다고 수정 권한이 생기지 않는다. |
| "검증은 나중에 /h-verify에서 하면 되지" | 실행 단계에서 검증하지 않으면 버그가 누적된다. |
| "대충 동작하니까 됐다" | 동작 ≠ 완료. 스펙과 테스트가 기준이다. |
| "이건 스펙보다 더 나은 방법이야" | 실행 단계에서 설계를 변경하지 마라. 의견이 있다면 /plan 수정부터. |
| "빨리 끝내야 하니까 품질을 줄이자" | 품질을 줄이면 나중에 3배 더 걸린다. |
| "5분이면 고칠 수 있는데 굳이 /plan까지 갈 필요 없어" | 5분이 30분이 되고 regression이 생긴다. /fix를 써라. |
| "이 테스트는 사소해서 건너뛰자" | 사소한 테스트에서 critical 버그가 잡힌다. |
| "인접 코드 리팩토링은 같이 하면 되지" | 한 번에 두 가지 하면 둘 다 버그가 난다. |

## Red Flags

다음 표현이 나오면 즉시 멈추고 자가 점검:
- "간단해서" / "it's simple"
- "나중에" / "later"
- "이미 열었으니까" / "already opened"
- "대충" / "roughly"
- "껌값이야" / "trivial"
- "시간 없으니까" / "no time"

## Violation Response

합리화가 감지되면 해당 단계를 중지하고, 규칙을 준수하며 재시작한다.
